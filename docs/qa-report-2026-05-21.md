# QA Report - 2026-05-21

## Context

Project: Bihar Board Class 10 Science AI Tutor  
Date: 2026-05-21  
Role: End-to-end QA review

This QA pass tested the current MVP across documentation, backend RAG scripts, vector retrieval, MongoDB-backed Ask flow, HTTP endpoints, and frontend browser behavior.

The core product rule for this project is:

> Zuno must answer only from retrieved/indexed approved study content. If retrieved content is not enough, it must clearly say that the available material does not contain the answer.

## Overall Summary

The lower-level RAG foundation is healthy. Loader, chunker, study map, curriculum resolver, vector-store validation, retrieval smoke tests, MongoDB model tests, Ask DB integration, and frontend production build all passed.

The main risks are in the new LLM-first Ask flow and the live frontend experience:

- Provider failures are not handled gracefully.
- Broad foundation questions can still produce unsupported general answers.
- The frontend can stay stuck in a loading state when the backend is slow or rate-limited.
- Regression coverage became thinner after the old deterministic conversation tests were removed.
- Some accessibility and frontend console issues exist.

## Findings

## P0 - LLM Provider Failure Is Not Graceful

### What Happened

During a live Ask E2E run, the `focus_ohm` case failed with a Groq rate-limit error:

```text
429 rate_limit_exceeded
Rate limit reached for model llama-3.3-70b-versatile
```

The request took around 94 seconds before failing. The raw provider error surfaced instead of a clean tutor-safe response.

### Why This Matters

This is a production-blocking issue because the app depends on LLM calls. Any provider outage, quota limit, timeout, invalid API key, or transient network problem can break the student experience.

For a student, the app should say something like:

```text
Abhi Zuno answer generate nahi kar pa raha. Thodi der baad try karo.
```

It should not expose provider internals, organization IDs, raw quota details, or stack-like technical errors.

### Likely Cause

The provider call in `generateTutorResponse()` is not wrapped in a safe fallback:

```text
backend/src/tutor/llmFlow/tutorResponder.js
```

Also, the global error handler returns `error.message` directly:

```text
backend/src/middlewares/error.middleware.js
```

### Possible Solutions

#### Solution A: Catch LLM Errors In Tutor Responder

Wrap the tutor LLM call in `try/catch`. If the provider fails, return a safe structured response.

Example behavior:

```json
{
  "status": "temporarily_unavailable",
  "responseMode": "conversation",
  "sections": [
    {
      "heading": "Thoda issue aa gaya",
      "content": "Abhi Zuno answer generate nahi kar pa raha. Thodi der baad try karo."
    }
  ]
}
```

Tradeoffs:

- Pros: Fastest fix, protects frontend immediately.
- Pros: Keeps response contract stable.
- Cons: Does not distinguish rate limit vs timeout vs malformed JSON unless error classification is added.

#### Solution B: Add Provider Error Classification

Create a small helper that maps provider failures to internal categories:

- `rate_limited`
- `timeout`
- `auth_error`
- `network_error`
- `invalid_model_response`
- `unknown_provider_error`

Then return safe user-facing messages while logging technical details server-side.

Tradeoffs:

- Pros: Better debugging and analytics later.
- Pros: Can show different retry guidance for rate limits vs auth errors.
- Cons: More code than Solution A.
- Cons: Provider error shapes may change.

#### Solution C: Add Fallback Provider

If Groq fails, fallback to another configured provider/model.

Tradeoffs:

- Pros: Better uptime.
- Pros: Useful during quota exhaustion.
- Cons: More cost and configuration complexity.
- Cons: Different models may produce different answer style.
- Cons: Needs careful prompt and grounding QA for every fallback model.

#### Recommended Approach

Start with Solution A plus basic error classification from Solution B. Do not add fallback provider yet unless the app needs higher uptime during MVP demos.

## P0 - Broad Foundation Questions Can Break Grounding Rule

### What Happened

The question:

```text
Science kya hai?
```

returned an answer like a general science definition. Retrieval returned some sources, but those sources are from chapter content and do not truly contain a curated foundation explanation of "Science kya hai?".

The answer status was `answered`, not `insufficient_context`.

### Why This Matters

This directly affects the project's core rule. Zuno should not answer from general knowledge when approved source content is weak or missing.

This is especially risky for broad questions:

- Science kya hai?
- Physics kya hai?
- Chemistry kya hai?
- Biology kya hai?
- Main padhta hoon par yaad nahi rehta.
- Science kaise padhein?

### Likely Cause

The current curated dataset contains chapter-level Science content, but not foundation/orientation content. The LLM receives some retrieved context and may still answer broadly from its own knowledge.

### Possible Solutions

#### Solution A: Add Curated Foundation Markdown

Add approved content under:

```text
data/class-10/science/foundation/
```

Recommended files:

```text
01-science-orientation.md
02-subject-orientation.md
03-study-skills.md
04-learning-support.md
```

Then rebuild vector store:

```bash
cd backend
npm.cmd run rag:index
```

Tradeoffs:

- Pros: Best match for product rule.
- Pros: Lets Zuno answer foundation and study-support questions safely.
- Cons: Requires content writing and review.
- Cons: Increases indexing scope.

#### Solution B: Strengthen Retrieval Sufficiency Check

Before calling the answer LLM, inspect retrieved chunks for relevance. If the query asks a broad foundation question but chunks are only weakly related, return `insufficient_context`.

Tradeoffs:

- Pros: Prevents unsupported answers even before new content is added.
- Pros: Protects all future missing-content cases.
- Cons: Relevance thresholds can be hard to tune.
- Cons: May refuse some answerable questions if the check is too strict.

#### Solution C: Ask LLM To Judge Grounding Before Answering

After retrieval, call a small "grounding judge" prompt:

```text
Does this context contain enough information to answer the student question?
Return yes/no and short reason.
```

Tradeoffs:

- Pros: More flexible than keyword thresholds.
- Pros: Can catch semantically weak retrieval.
- Cons: Adds another LLM call and latency.
- Cons: Judge itself can be wrong.

#### Solution D: Add Topic Allowlist For Foundation Questions

Detect foundation questions with rules and refuse unless foundation content exists.

Tradeoffs:

- Pros: Simple and predictable.
- Pros: Good temporary guardrail.
- Cons: Can become brittle.
- Cons: Adds manual intent logic back into an LLM-first architecture.

#### Recommended Approach

Use Solution A and Solution B together. Add curated foundation content, then add a lightweight sufficiency gate so future missing topics fail safely.

## P1 - Frontend Can Stay Stuck In Loading State

### What Happened

In the browser, submitting `hii` showed:

```text
Zuno is preparing an answer
```

After 37+ seconds, the UI was still waiting and no clear error message was shown.

### Why This Matters

Students may think the app is broken. A tutor app should give quick feedback when the backend is slow, rate-limited, or unavailable.

### Likely Cause

`fetch()` calls in the frontend do not use timeout or abort logic:

```text
frontend/src/api/tutorApi.js
```

The frontend waits until the browser/network stack finishes.

### Possible Solutions

#### Solution A: Add Frontend Request Timeout

Use `AbortController` with a 30-45 second timeout for `askTutor()`.

Tradeoffs:

- Pros: Simple and user-friendly.
- Pros: Prevents infinite loading.
- Cons: Backend may still finish later and save history, while frontend has already timed out.

#### Solution B: Add Backend Timeout Around LLM Calls

Use a server-side timeout for decider and responder calls.

Tradeoffs:

- Pros: Prevents server resources from hanging.
- Pros: Gives consistent error response to all clients.
- Cons: Needs careful timeout values because embeddings and LLM calls can be slow.

#### Solution C: Use Streaming Or Progress Events Later

Stream partial states:

- deciding
- retrieving
- generating
- final answer

Tradeoffs:

- Pros: Best UX long-term.
- Pros: Student sees progress.
- Cons: More implementation complexity.
- Cons: Not needed for the thin MVP yet.

#### Recommended Approach

Add frontend timeout now and backend timeout soon after. Keep streaming as a later enhancement.

## P1 - Conversation Regression Coverage Is Too Thin

### What Happened

Old deterministic scripts were removed:

```text
backend/scripts/test-lesson-flow.js
backend/scripts/test-tutor-conversations.js
```

Current `test:ask-db` checks session/history persistence using a simple greeting, but it does not deeply test the LLM-first behavior.

### Why This Matters

The LLM-first flow is more flexible, but also more unpredictable. Without regression tests, small prompt or code changes can silently break:

- grounding behavior
- out-of-scope handling
- focus mode behavior
- source attachment
- language style
- follow-up behavior

### Possible Solutions

#### Solution A: Add Mocked LLM Regression Tests

Mock decider and tutor responder outputs, then test Ask service behavior deterministically.

Tradeoffs:

- Pros: Fast and reliable.
- Pros: Good for persistence, validation, source handling, response shape.
- Cons: Does not test real prompt quality.

#### Solution B: Add Live LLM Smoke QA Script

Create a script with a small set of live prompts:

- `hii`
- `electric current kya hota hai?`
- `Science kya hai?`
- `mujhe cricket team bna do`
- focus mode `ohm law kya hai?`

Tradeoffs:

- Pros: Tests real model behavior.
- Pros: Catches prompt regressions.
- Cons: Slow, flaky, quota-dependent.
- Cons: Should not be required for every local test run.

#### Solution C: Add Golden Snapshot Evaluation

Save expected high-level properties, not exact wording:

- status should be `answered`
- sources length should be greater than 0
- answer should be Roman Hinglish
- unsupported foundation should be `insufficient_context`

Tradeoffs:

- Pros: Better fit for LLM variability.
- Pros: Easier to maintain than exact answer snapshots.
- Cons: Requires custom assertions.

#### Recommended Approach

Use Solution A for normal regression and Solution B/C for occasional live QA before demos or releases.

## P1 - Ask API Does Not Fully Protect Against Provider Latency

### What Happened

The live E2E case that hit rate limits took around 94 seconds before returning an error.

### Why This Matters

Long waits make the app feel broken and can keep frontend state locked.

### Possible Solutions

#### Solution A: Add Per-Step Timers

Measure time for:

- DB session/state loading
- decider LLM
- retrieval
- responder LLM
- DB save

Tradeoffs:

- Pros: Shows actual bottleneck.
- Cons: Only observability; does not fix latency by itself.

#### Solution B: Hard Timeout LLM Calls

Set max time per LLM call, for example 20-30 seconds.

Tradeoffs:

- Pros: Prevents long hangs.
- Cons: Can fail legitimate slow answers.

#### Solution C: Cache Vector Store And Clients

Avoid reloading expensive resources per request where possible.

Tradeoffs:

- Pros: Improves average latency.
- Cons: Requires cache invalidation after reindex.

#### Recommended Approach

Start with timing logs and hard LLM timeout. Then optimize caching after measuring.

## P2 - Frontend Console Warning For alignItems Prop

### What Happened

Browser console showed:

```text
React does not recognize the `alignItems` prop on a DOM element.
```

### Why This Matters

This does not currently break the app, but console errors reduce confidence and can hide more serious issues later.

### Possible Solutions

#### Solution A: Inspect MUI Imports And Component Prop Forwarding

Check components using `Stack alignItems`, especially:

```text
frontend/src/components/AppHeader.jsx
frontend/src/components/FocusModal.jsx
frontend/src/components/Sidebar.jsx
```

Make sure `Stack` is actually MUI Stack and not being transformed into a custom DOM component somewhere.

Tradeoffs:

- Pros: Fixes root cause.
- Cons: Needs quick component inspection.

#### Solution B: Move Layout Props Into sx

Instead of:

```jsx
<Stack direction="row" alignItems="center">
```

use:

```jsx
<Stack sx={{ flexDirection: 'row', alignItems: 'center' }}>
```

Tradeoffs:

- Pros: Avoids prop leakage if a component wrapper is involved.
- Cons: Less idiomatic MUI.

#### Recommended Approach

First inspect the exact component causing the warning. If unclear, move only the offending instance to `sx`.
